# Competency Co-pilot — 能力模型研讨会工具 开发总结

> 最后更新: 2026-03-29
> 适用场景: 2026-03-31 数字化中心能力模型共创研讨会

---

## 一、项目背景与目标

### 1.1 业务背景

为"数字化中心能力模型共创研讨会"开发交互式研讨工具。研讨会需要让业务专家、培训中心、咨询顾问三方共同评审 AI 从 7 份源材料中提取的能力模型，通过多轮研讨确认最终的能力项、二级能力、关键行为及学习资源。

### 1.2 核心目标

| 目标 | 说明 |
|------|------|
| AI 能力提取 | 从访谈汇总、测评数据、外部对标等源材料中自动提取能力项 |
| 多方研讨 | 提供交互式工具支持专家评审、讨论、确认 |
| 数据可视化 | 雷达图、胶囊池、三栏浏览器等直观展示能力结构 |
| 学习资源 | AI 为每个能力项生成匹配的学习资源推荐 |

## 三、AI 数据管线

### 3.1 预计算流程 (precompute.py)

```
源材料 (7份, ~45,000字)
        │
        ▼
  ┌─────────────┐
  │   FR-1.2    │  验证种子 + 发现新能力
  │ 验证与发现   │  17 种子验证 + 32 新发现 = 49 项
  └─────┬───────┘
        │
        ▼
  ┌─────────────┐
  │   FR-1.3    │  二级展开
  │  二级展开    │  49 × 5 = 245 个二级能力项
  └─────┬───────┘
        │
        ▼
  ┌─────────────┐
  │   FR-1.4    │  行为展开
  │  行为展开    │  245 × ~5 = 1,260 个关键行为
  └─────┬───────┘
        │
        ▼
  ┌─────────────┐
  │   FR-1.5    │  学习资源生成
  │  资源生成    │  AI 为每个一级能力项推荐学习资源
  └─────┬───────┘
        │
        ▼
  ┌─────────────┐
  │  权重校准    │  人工权重 × AI 证据强度
  │ adjust_      │  秒级完成, 无需重跑 AI
  │ weights.py  │
  └─────┬───────┘
        │
        ▼
  result.json (最终输出)
```

| 阶段 | 调用次数 | 模型 | 产出 |
|------|----------|------|------|

| FR-1.3 二级展开 | 49 次 (按 L1) | qwen-plus | 245 二级项 |
| FR-1.4 行为展开 | 245 次 (按 L2) | qwen-plus | 1,260 行为 |

### 3.4 权重校准机制

```
综合评分 = AI 证据强度 × 人工权重
```

人工权重基于三方评估:
- **公司与部门需求**: 岗位核心职责界定、能力项与岗位匹配度
- **培训中心意见**: 可培训性评估、课程覆盖与资源匹配
- **咨询顾问评估**: 行业对标与最佳实践、能力模型结构合理性

权重示例:

| 能力项 | AI 证据强度 | 人工权重 | 综合评分 | 说明 |
|--------|-----------|---------|---------|------|
| 业务需求转化 | 0.96 | 1.0 | 0.96 | 种子项 · 核心职责 |
| 全栈实现能力 | 0.94 | 1.0 | 0.94 | 种子项 · 核心职责 |
| AI技术理解与应用 | 0.92 | 1.0 | 0.92 | 新晋升种子项 |
| 翻译官能力 | 0.73 | 0.75 | 0.55 | 角色描述 · 非能力维度 |
| 业务洞察与引领能力 | 0.49 | 0.50 | 0.25 | 跨模型 · 非核心职责 |

### 3.5 种子能力列表

**交付管理 (9 项)**:

1. 技术架构能力 — 以前瞻性的系统化思维，规划与设计稳定、可扩展的技术系统蓝图的能力
2. 全栈实现能力 — 运用广博而深入的工程技术，将技术蓝图高质量地转化为可靠软件产品的能力
3. 工程化交付实施 — 通过建立标准化、自动化的流程与工具，保障系统开发过程高效、规范、可控的能力
4. 系统稳定性保障 — 通过系统化的问题解决和主动的风险规避，确保线上系统长期稳定运行的能力
5. 业务需求转化 — 准确理解业务诉求，并将其精准地转化为清晰、可执行的技术任务与规范的能力
6. 技术创新驱动 — 主动探索并应用前沿技术，为现有业务场景赋能并创造新价值的能力
7. 技术团队管理 — 识别、发展并激励技术人才，打造高绩效、高凝聚力技术团队的能力
8. 跨方技术协同 — 在多技术栈、多利益方的复杂环境中，推动技术层面的共识与顺畅协作的能力
9. AI技术理解与应用 — 了解AI Agent开发的概念与方法，能利用主流AI辅助开发手段进行开发、测试和维护 (新增)

**项目/业务管理 (8 项)**:

1. 业务领域精通 — 深刻理解业务领域的运作模式、流程脉络、痛点及战略方向的专业能力
2. 业务持续性保障 — 通过快速的问题解决和系统优化，保障并提升现有业务流程的稳定性与运行效率的能力
3. 整合项目交付 — 运用专业知识和工具，对项目从启动到收尾的全过程进行整合性管理的能力
4. 解决方案设计 — 将模糊的业务问题，解构并设计为清晰、合理、可落地的数字化解决方案的能力
5. 数据驱动决策 — 利用数据分析洞察业务本质、评估项目成效，并以此驱动业务与项目决策的能力
6. 需求与价值实现 — 挖掘、引导并管理与战略对齐的业务需求，并推动其端到端落地以实现可衡量商业价值的能力
7. 团队管理与赋能 — 凝聚、激励并发展跨职能项目团队，营造积极协作的氛围的能力
8. 协同与影响 — 在复杂的组织环境中，与所有内外部干系人建立信任，推动协同的能力

## 四、前端功能详解

### 4.1 四 Tab 布局

研讨会页面 (`/workshop/competency`) 包含四个选项卡:

```
┌──────────┬──────────────┬──────────────┬──────────────┐
│  分析流程  │ 一级能力项研讨 │ 二级与行为探索 │ 学习资源盘点  │
└──────────┴──────────────┴──────────────┴──────────────┘
```

### 4.2 Tab 1: 分析流程 (DataFlowDiagram)

**功能**: 可视化展示 AI 分析的完整流程, 让研讨会参与者理解数据来源和处理逻辑。

**5 个阶段**:
1. **源材料输入** — 展示 7 份源文件 (按格式 docx/xlsx/pptx 分色)
2. **种子能力列表** — 展示 17 个专家预设种子 (按模型分色)
3. **AI 三步分析管线** — FR-1.2 (验证+发现) → FR-1.3 (二级展开) → FR-1.4 (行为展开)
4. **综合权重校准** — 展示权重公式、三方权重来源、校准示例
5. **输出** — 三个研讨工具的概览

**技术特点**: 响应式设计, 移动端堆叠布局, 桌面端网格布局。

### 4.3 Tab 2: 一级能力项研讨 (CompetencyRadarView)

**功能**: 通过雷达图和胶囊池展示一级能力项, 支持种子/发现区分和专家确认。

**布局**:
```
┌─────────────────────────┬────────────┐
│                         │            │
│       雷达图              │  溯源面板   │
│  (Recharts RadarChart)   │ (悬停显示)  │
│                         │            │
├─────────────────────────┴────────────┤
│          备选池 (发现项胶囊)              │
├────────────────────────────────────────┤
│          专家确认区 (textarea)            │
└────────────────────────────────────────┘
```

**核心逻辑**:
- 雷达图仅显示种子项 (交付 9 项, 业务 8 项)
- 雷达图下方显示发现项胶囊池
- 悬停雷达点或胶囊时, 右侧溯源面板显示详细信息 (名称、定义、证据强度、来源文件)
- 模型切换按钮 (交付管理 / 项目业务管理)
- 保存确认结果到后端

**动态雷达大小**: 根据种子数量自动调整, 不固定为 8 项。

### 4.4 Tab 3: 二级能力与关键行为探索 (CompetencyExplorer)

**功能**: 三栏联动浏览器, 逐级展开能力结构。

**布局**:
```
┌──────────┬──────────┬──────────┐
│ 一级能力项  │ 二级能力项  │  关键行为   │
│ (25/24项) │ (每L1约5项)│ (每L2约5项)│
│          │          │          │
│ 悬停显示   │ 名称下方   │ 按层级分组  │
│ 定义tooltip│ 显示定义    │ 初/中/高   │
└──────────┴──────────┴──────────┘
```

**核心逻辑**:
- 左栏: 全部能力项 (种子在前, 发现在后), 按分数降序
- 中栏: 选中 L1 后展开其二级能力项, 每个项下方显示一句话定义
- 右栏: 选中 L2 后展开关键行为, 按层级 (初级/中级/高级) 分组
- 层级定义: 初级=在他人指导下工作, 中级=独立工作, 高级=赋能他人
- 底部: 确认区 (二级能力项 textarea + 关键行为 textarea)

### 4.5 Tab 4: 学习资源盘点 (ResourceInventory)

**功能**: 完整能力模型树 + AI 推荐学习资源清单。

**布局**:
```
┌──────────────────┬──────────────────┐
│   能力模型树       │   学习资源列表     │
│ (带资源数量标记)    │ (按类型/层级展示)  │
│                  │                  │
│ L1 > L2 > 行为    │ 📕 书籍           │
│                  │ 🎬 视频           │
│                  │ 💻 在线课程        │
└──────────────────┴──────────────────┘
┌────────────────────────────────────────┐
│            备注 (textarea)              │
└────────────────────────────────────────┘
```

**资源类型**: 书籍、视频、在线课程、案例库、工具、模板
**适用层级**: 初级 (在他人指导下工作)、中级 (独立工作)、高级 (赋能他人)

### 5.2 关键技术决策

#### 决策 1: 数据预处理 vs 实时 AI 调用

**选择**: 数据预处理 (预计算 result.json)

**理由**: 研讨会需要稳定、快速的数据展示。预计算后前端只需加载一个 JSON 文件, AI 调用在会前完成。

#### 决策 2: 权重校准方案

**选择**: 乘法权重 (AI 分 × 人工权重), 在预计算结果上直接应用

**理由**: 除法权重需要记录原始 AI 分并重跑计算, 乘法权重只需一个简单脚本秒级完成。

**重要**: 脚本实现了权重反转机制 — 如果已有旧权重, 先除以旧权重恢复原始分, 再乘以新权重, 避免多次运行导致权重累积。

#### 决策 3: SSR 预渲染策略

**选择**: `export const dynamic = 'force-dynamic'` 在 Dashboard 布局层

**理由**: 所有 Dashboard 页面都是认证后动态页面, 静态预渲染没有价值。客户端库 (recharts, reactflow) 在模块级别访问 `window`, 导致预渲染失败。在布局层禁用预渲染一次性解决所有页面问题。

#### 决策 4: 前后端分离 vs 一体化

**选择**: 前后端分离 (Vercel + Docker)

**理由**: 前端 Vercel CDN 加速, 后端 Docker 独立部署。数据通过 REST API 传输, result.json 通过 volume mount 挂载到容器。

#### 决策 5: 种子能力晋升机制

**选择**: 通过 adjust_weights.py 的 SEED_PROMOTIONS 字典

**触发**: 阅读《Operating System Concepts》和《Team Topology》后, 识别出 AI 技术理解与应用应为核心能力。

**实现**:
1. 在 seeds.json 中添加新种子
2. adjust_weights.py 自动设置 weight=1.0 并更新 origin='seed'
3. 去重: 跨模型重复项保留数据更丰富的版本

### 5.3 解决的关键问题

| 问题 | 根因 | 解决方案 |
|------|------|----------|
| Vercel 构建失败 "window is not defined" | recharts/reactflow 模块级访问 window | layout.tsx 添加 dynamic='force-dynamic' |
| 手机端侧边栏关不掉 | 冲突的 CSS translate 类 | 分离 mobile/desktop 控制逻辑 |
| 雷达图固定 8 项 | 硬编码 RADAR_SIZE=8 | 动态计算 seedCount |
| 权重多次运行累积 | 乘法权重无反转机制 | 先除旧权重再乘新权重 |
| 跨模型重复项 | 晋升种子时产生副本 | 去重保留 secondary_terms 更多版本 |
| Explorer 只显示种子项 | confirmedTerms 逻辑限制 | 改为按模型显示全部 49 项 |

### 7.1 能力模型规模

| 维度 | 交付管理 | 项目/业务管理 | 合计 |
|------|---------|-------------|------|

### 7.2 行为层级分布

每个二级能力项包含 5 个关键行为, 分布在三个层级:
- **初级** (在他人指导下工作): ~1-2 条
- **中级** (独立工作): ~2-3 条
- **高级** (赋能他人): ~1-2 条

## 八、使用指南

### 8.1 研讨会流程

1. **开场** — 切换到"分析流程"Tab, 向参与者展示 AI 分析全过程
2. **一级能力评审** — 切换到"一级能力项研讨"Tab:
   - 查看雷达图中的种子项
   - 评估备选池中的发现项
   - 在确认区记录最终确认的一级能力列表
3. **二级能力探索** — 切换到"二级与行为探索"Tab:
   - 逐个一级项展开二级能力
   - 评审关键行为描述
   - 在确认区记录调整
4. **资源盘点** — 切换到"学习资源盘点"Tab:
   - 查看每个能力的 AI 推荐资源
   - 记录资源匹配差距

### 8.2 快捷操作

- **H 键**: 切换侧边栏 (演示模式, 隐藏侧边栏获得更大展示空间)
- **模型切换**: 每个Tab都有交付管理/项目业务管理切换按钮

## 九、已知限制与后续优化

| 限制 | 说明 |
|------|------|
| 数据非实时 | result.json 需手动同步到服务器 |
| 定义为 AI 生成 | L1/L2 定义质量依赖 AI, 可能需要人工润色 |
| 无离线模式 | 依赖后端 API, 无法离线使用 |
| 单人编辑 | 确认区为 textarea, 无多人协作冲突处理 |
| 资源真实性 | AI 推荐的资源需人工验证是否存在 |
