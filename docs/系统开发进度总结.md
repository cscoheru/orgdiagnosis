## 一、项目概述

### 1.1 项目背景

将现有的"五维组织诊断工具"升级为"咨询建议书智能协同生成系统"，实现：
- 诊断数据 + 历史咨询知识库 + 互联网信息的深度融合
- Human-in-the-loop 人工干预机制
- Gamma 级别的 PPTX 报告导出

### 1.2 核心功能

1. **结构化需求录入**: 通过4步向导表单收集客户需求
2. **智能大纲生成**: 基于LlamaIndex RAG检索历史报告，AI生成报告大纲
3. **人工审核干预**: 用户可编辑修改大纲和内容
4. **PPTX导出**: 专业咨询报告格式输出

### 1.3 技术栈

| 层级 | 技术 | 版本 | 用途 |
|------|------|------|------|
| 前端框架 | Next.js | 16.1.7 | App Router架构 |
| UI框架 | React | 19.2.3 | 组件化开发 |
| 样式 | Tailwind CSS | 4.x | 响应式设计 |
| 后端框架 | FastAPI | 0.110+ | RESTful API |
| RAG引擎 | LlamaIndex | 0.10+ | 知识库检索 |
| 向量存储 | ChromaDB | - | 本地向量数据库 |
| 工作流引擎 | LangGraph | - | 异步任务编排 |
| Embedding | DashScope | text-embedding-v3 | 向量化 |
| PPT生成 | python-pptx | 0.6.23 | PPTX渲染 |
| 测试框架 | Playwright | 1.58+ | E2E测试 |

## 二、开发阶段详情

### Phase 1: LlamaIndex 知识库构建 ✅ 完成

**开发周期**: 2天
**完成日期**: 2026-03-18

#### 2.1.1 目标
建立历史咨询报告的向量化检索能力，支持按MDS五维度检索相关素材。

#### 2.1.2 实现内容

**文件结构**:
```
backend/lib/llamaindex/
├── __init__.py              # 模块入口
├── document_processor.py    # 文档解析器
├── indexer.py               # 索引构建器
├── retriever.py             # 语义检索器
└── embeddings.py            # 自定义Embedding封装
```

**核心功能**:

1. **多格式文档解析** (`document_processor.py`)
   - 支持 PDF、DOCX、PPTX 格式
   - 使用 `HierarchicalNodeParser` 进行层级分割
   - 自动推断文档类别（战略/HR/绩效/薪酬/人才）

2. **向量索引构建** (`indexer.py`)
   - 使用 DashScope `text-embedding-v3` 模型
   - ChromaDB 持久化存储到 `backend/storage/chroma/`
   - 支持增量更新

3. **语义检索** (`retriever.py`)
   - 按MDS五维度关键词映射
   - 支持元数据过滤
   - Top-K 相似度检索

4. **批处理Embedding** (`embeddings.py`)
   - 解决 DashScope API 批次限制（10个/批）
   - 实现重试逻辑 + 指数退避
   - 批次间300ms延迟防止限流

#### 2.1.3 知识库统计

| 指标 | 数值 |
|------|------|
| 历史报告数量 | 18份 |
| 文档块数量 | 610个 |
| 分类目录 | strategy, hr, performance, compensation, talent |
| 向量维度 | 1024 |

#### 2.1.4 解决的问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| DashScope batch size限制 | API最多处理10个文本 | 自定义BatchedDashScopeEmbedding包装器 |
| PyMuPDFReader API变更 | `load_data(file=...)` 参数已弃用 | 改用 `load_data(Path(file))` |
| ChromaDB元数据验证 | 不支持list/dict类型 | `_sanitize_metadata()` 清洗函数 |
| Pydantic v2 ClassVar | 需要显式类型注解 | `ClassVar[int]` + `PrivateAttr()` |

### Phase 2: LangGraph 交互式工作流 ✅ 完成

**开发周期**: 3天
**完成日期**: 2026-03-19

#### 2.2.1 目标
实现带人工干预断点的异步任务流，支持暂停、恢复、状态持久化。

#### 2.2.2 实现内容

**文件结构**:
```
backend/lib/report_workflow/
├── __init__.py              # 模块入口
├── state.py                 # 状态定义
├── nodes.py                 # 工作流节点函数
└── workflow.py              # 工作流管理器
```

**工作流状态机**:
```
pending → generating_outline → outline_ready → [人工确认]
       → generating_slides → slides_ready → [人工确认]
       → ready_for_export → completed
```

**API端点**:
```
POST /api/requirement/template    # 获取需求模板
POST /api/requirement/validate    # 验证需求数据
POST /api/report/start            # 启动报告生成
GET  /api/report/status/{id}      # 查询任务状态
GET  /api/report/outline/{id}     # 获取生成的大纲
POST /api/report/confirm-outline  # 确认大纲
GET  /api/report/slides/{id}      # 获取幻灯片内容
POST /api/report/confirm-slides   # 确认幻灯片
GET  /api/report/export/{id}      # 下载PPTX
DELETE /api/report/task/{id}      # 取消任务
```

#### 2.2.3 数据契约

**ClientRequirement (输入)**:
```python
class ClientRequirement(BaseModel):
    client_name: str                              # 客户名称
    industry: IndustryType                        # 行业类型
    industry_background: str = Field(min_length=50)  # 行业背景
    company_intro: str = Field(min_length=50)     # 公司介绍
    core_pain_points: List[str]                   # 核心痛点
    project_goals: List[str]                      # 项目目标
    phase_planning: List[ProjectPhase]            # 阶段规划
    main_tasks: List[str]                         # 主要任务
    deliverables: List[str]                       # 交付成果
```

**ReportDraft (输出)**:
```python
class ReportDraft(BaseModel):
    report_id: str
    client_name: str
    part1_understanding: ReportSection      # 项目需求的理解
    part2_methodology: ReportSection        # 项目方法与整体框架
    part3_implementation: ReportSection     # 项目实施步骤
    part4_plan_team_pricing: ReportSection  # 项目计划、团队与报价
```
